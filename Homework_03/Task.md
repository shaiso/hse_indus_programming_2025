# НИС — Итоговый проект по дисциплине

## Формат проекта

Итоговый проект представляет собой работающий сервис или приложение, использующее модели машинного обучения (ML или нейросети).

**Примеры форматов:**

- Web-сервис
- REST API
- Telegram / Discord бот
- CLI-инструмент
- AI-агент
- Сервис обработки изображений / текста / аудио / документов
- NLP-система (NER, классификация, поиск, суммаризация и др.)

> Проект засчитывается как **4 домашних работы** и оценивается по четырем блокам.

---

## Обязательные минимальные требования

Проект **не может быть принят**, если отсутствует хотя бы один из пунктов:

- [ ] Рабочий запуск (инструкция + воспроизводимость)
- [ ] Использование ML-модели с обоснованием выбора
- [ ] Метрики качества и их интерпретация
- [ ] Архитектурная схема проекта
- [ ] Формализованные требования и описание MVP
- [ ] Демонстрация работы

---

## Структура оценивания (100 баллов)

| Блок | Баллы |
|------|-------|
| 1. Реализация | 25 |
| 2. Техническая корректность | 25 |
| 3. Архитектура | 25 |
| 4. Требования и MVP | 25 |

---

### Блок 1 — Реализация и готовность решения (25 баллов)

#### Что должно быть реализовано

**1. Рабочий продукт**

- [ ] Проект запускается по инструкции
- [ ] Нет критических ошибок
- [ ] Реализован основной пользовательский сценарий
- [ ] Есть интерфейс (API / UI / бот / CLI)

**2. Репозиторий**

- [ ] README с:
  - описанием проекта
  - инструкцией по запуску
  - описанием архитектуры
- [ ] `requirements.txt` / `pyproject.toml`
- [ ] Пример входных данных
- [ ] Описание переменных окружения

**3. Демонстрация**

- [ ] Видео или live-демо
- [ ] Показан основной флоу

#### Критерии оценки

| Баллы | Описание |
|-------|----------|
| 0–10 | Частично работает, нет стабильности, нет инструкции |
| 11–18 | Работает основной сценарий, есть документация, есть минимальный интерфейс |
| 19–25 | Полностью рабочий сервис, удобный запуск (Docker приветствуется), понятная структура, обработка ошибок |

---

### Блок 2 — Техническая корректность и инженерное качество (25 баллов)

#### Что обязательно должно быть

**1. Работа с моделью**

- [ ] Обоснован выбор модели
- [ ] Описаны ограничения
- [ ] Реализована предобработка и постобработка
- [ ] Обработка ошибок

**2. Метрики качества**

- [ ] Выбрана релевантная метрика
- [ ] Описан датасет
- [ ] Есть замеры на тестовой выборке
- [ ] Есть интерпретация результата

**Для LLM-проектов дополнительно:**

- [ ] Замер latency
- [ ] Оценка стоимости (если API)
- [ ] Поведение на длинных запросах

**3. Производительность**

- [ ] Замер времени ответа
- [ ] Базовый нагрузочный тест
- [ ] Анализ узких мест

**4. Тестирование**

- [ ] Минимум 3–5 unit-тестов
- [ ] Проверка критических функций
- [ ] Проверка невалидных данных

#### Критерии оценки

| Баллы | Описание |
|-------|----------|
| 0–10 | Метрики отсутствуют или формальные, нет понимания модели, нет тестов |
| 11–18 | Есть метрики и замеры, базовые тесты, частичное понимание ограничений |
| 19–25 | Глубокий анализ качества, обоснованный выбор модели, нагрузочное тестирование, понимание trade-off'ов |

---

### Блок 3 — Архитектура и расширяемость (25 баллов)

#### Обязательные элементы

**1. Архитектурная схема**

- [ ] Диаграмма (блок-схема / C4)
- [ ] Выделены слои (API, бизнес-логика, ML, инфраструктура)

**2. Структура проекта**

- [ ] Разделение ответственности
- [ ] Отсутствие "монолитного скрипта"
- [ ] Возможность заменить модель

**3. Интеграция**

- [ ] Есть API или возможность интеграции
- [ ] Конфигурация вынесена в настройки
- Плюсом будет: Docker, OpenAPI, docker-compose

**4. Масштабируемость** (в письменном виде)

- [ ] Где узкое место?
- [ ] Как масштабировать?
- [ ] Какие риски?

#### Критерии оценки

| Баллы | Описание |
|-------|----------|
| 0–10 | Один большой файл, нет схемы, нет разделения слоев |
| 11–18 | Есть базовая архитектура, можно расширять, есть диаграмма |
| 19–25 | Четкое разделение слоев, расширяемость, готовность к продакшену, понимание масштабирования |

---

### Блок 4 — Работа с требованиями и MVP (25 баллов)

#### Обязательные элементы

**1. Формализация задачи**

- [ ] Описание проблемы
- [ ] Целевая аудитория
- [ ] Пользовательские сценарии
- [ ] Ограничения
- [ ] Нефункциональные требования

**2. MVP**

- [ ] Выделен минимальный функционал
- [ ] Обоснование выбора функций
- [ ] Что НЕ вошло и почему

**3. Обоснование ML**

- [ ] Почему используется ML?
- [ ] Почему эта модель?
- [ ] Можно ли решить без ML?

**4. План развития**

- [ ] 3 направления развития
- [ ] Риски проекта

#### Критерии оценки

| Баллы | Описание |
|-------|----------|
| 0–10 | Нет формализации, MVP не выделен |
| 11–18 | Есть требования, есть описание аудитории, есть MVP |
| 19–25 | Четкое product-мышление, осознанный выбор ML, реалистичный roadmap |

## Итоговый чек-лист перед сдачей

**Техническая часть:**

- [ ] Проект запускается
- [ ] Есть инструкция
- [ ] Есть метрики
- [ ] Есть тесты
- [ ] Есть замеры производительности

**Архитектура:**

- [ ] Есть диаграмма
- [ ] Проект структурирован
- [ ] Можно заменить модель

**Продуктовая часть:**

- [ ] Есть формализация задачи
- [ ] Есть MVP
- [ ] Есть roadmap

**Демонстрация:**

- [ ] Рабочий сценарий
- [ ] Ответы на вопросы по архитектуре и метрикам

---

## Формат защиты

- **5 минут** — презентация
- **5 минут** — технические вопросы
- **5 минут** — вопросы по архитектуре и требованиям

Студент должен уметь объяснить:

- Почему выбран именно этот подход
- Какие есть ограничения
- Как проект масштабировать

---

> **Важно:** Использование ML должно быть осмысленным, а не формальным. Проект оценивается не по сложности модели, а по качеству инженерного и продуктового подхода.
