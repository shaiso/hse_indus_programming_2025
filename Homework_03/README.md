# AutoDoc API

REST API сервис на Go, который анализирует исходный код приложения и автоматически генерирует OpenAPI 3.0 спецификацию с описанием эндпоинтов, параметров и моделей данных.

## Проблема

Документирование API — рутинная и часто игнорируемая задача. Легаси-проекты остаются без документации, разработчики не пишут аннотации, а при онбординге новых членов команды приходится вручную разбираться в коде. Существующие инструменты (swag, go-swagger) требуют специальных комментариев в коде, а фреймворк-специфичные генераторы (FastAPI) работают только в своей экосистеме.

## Решение

AutoDoc API принимает исходный код сервиса и с помощью LLM автоматически:
- Определяет язык и фреймворк
- Извлекает эндпоинты, параметры, модели
- Генерирует валидную OpenAPI 3.0 спецификацию (YAML/JSON)
- Формирует человекочитаемое описание API

## Целевая аудитория

- Backend-разработчики, работающие с легаси-кодом без документации
- Тимлиды, которым нужно быстро понять API незнакомого сервиса
- Новые члены команды при онбординге

## Стек технологий

- **Язык:** Go
- **HTTP-фреймворк:** Gin
- **LLM:** OpenAI API (GPT-4.1-mini) — основная модель; GPT-4.1 — fallback для сложных случаев
- **Контейнеризация:** Docker, docker-compose
- **Тестирование:** стандартный пакет `testing` + testify
- **Линтер:** golangci-lint

## Архитектура

Проект — единый Go-бинарник, построенный по принципу чистой архитектуры с разделением на компоненты:

```
                    ┌───────────────────────────────────┐
                    │           Go Application          │
                    │                                   │
  HTTP Request ───▶ │  Middleware → HTTP Handlers       │
                    │                  │                │
                    │            Analyzer Service       │
                    │           /       |       \       │
                    │     Parser    LLM Client   Spec   │
                    │                  │        Builder │
                    └──────────────────│────────────────┘
                                       │
                                       ▼
                              ┌─────────────────┐
                              │  OpenAI API     │
                              │  (GPT-4.1-mini) │
                              └─────────────────┘
```

- **Middleware** — логирование, CORS, rate limiting, panic recovery
- **HTTP Handlers** — приём запросов, валидация, формирование ответа
- **Analyzer Service** — оркестрация: парсинг → LLM → сборка спеки
- **Code Parser** — определение языка/фреймворка, извлечение релевантных фрагментов
- **LLM Client** — формирование промптов, вызов OpenAI API, парсинг ответа
- **Spec Builder** — валидация OpenAPI 3.0 и форматирование в YAML/JSON

Подробная C4-диаграмма (3 уровня) — см. `docs/likec4/`.

## API

### `POST /api/v1/analyze`

Анализирует исходный код и возвращает OpenAPI-спецификацию.

**Request:**
```json
{
  "files": [
    {
      "name": "main.go",
      "content": "package main\n\nimport ..."
    }
  ],
  "language": "go",
  "format": "yaml"
}
```

**Response:**
```json
{
  "spec": "openapi: 3.0.0\ninfo:\n  title: ...",
  "summary": "Обнаружено 5 эндпоинтов: ...",
  "endpoints_count": 5,
  "format": "yaml"
}
```

### `GET /api/v1/health`

Проверка состояния сервиса.

## Обоснование использования ML

### Почему ML?

Извлечение API-структуры из произвольного кода — задача, которую невозможно решить набором регулярных выражений или статическим анализом для всех языков и фреймворков одновременно. LLM способна понимать семантику кода, распознавать паттерны маршрутизации разных фреймворков и генерировать структурированное описание.

### Выбор модели

При выборе LLM для анализа кода ключевые критерии:
- **Контекстное окно** — исходный код проекта может быть большим, нужно вместить несколько файлов
- **Качество работы с кодом** — модель должна понимать синтаксис разных языков и фреймворков
- **Стоимость** — сервис должен быть экономически обоснованным
- **Поддержка structured output** — нужен валидный JSON/YAML на выходе

#### Сравнение моделей

| Модель | Input / 1M tokens | Output / 1M tokens | Контекст | Качество кода |
|---|---|---|---|---|
| GPT-4.1 | $2.00 | $8.00 | 1M | Отличное |
| **GPT-4.1-mini** | **$0.40** | **$1.60** | **1M** | **Хорошее** |
| GPT-4.1-nano | $0.10 | $0.40 | 1M | Среднее |
| GPT-4o | $2.50 | $10.00 | 128K | Отличное |
| Claude Sonnet 4 | $3.00 | $15.00 | 200K | Отличное |
| Gemini 2.5 Flash | $0.15 | $0.60 | 1M | Хорошее |

#### Выбор: GPT-4.1-mini (основная) + GPT-4.1 (fallback)

**GPT-4.1-mini** выбрана как основная модель:
- **1M контекстное окно** — можно отправить весь проект целиком, не разбивая на части. Это критично для анализа кода, где связи между файлами важны для понимания API
- **$0.40 / $1.60 за 1M токенов** — в ~6 раз дешевле GPT-4o при сопоставимом качестве для задач анализа кода
- **Structured output** — нативная поддержка JSON mode для генерации валидных спецификаций
- Средний запрос (~5K input + ~2K output) обойдётся в ~$0.005 — менее 1 цента

**GPT-4.1** используется как fallback для сложных случаев (большие проекты, нестандартные фреймворки), где mini-версия не справляется с качеством.

**Почему не другие:**
- **GPT-4o** — контекст всего 128K, дороже, нет преимуществ для нашей задачи
- **Claude Sonnet 4** — отличное качество, но в 7.5x дороже по input и контекст 200K vs 1M
- **Gemini 2.5 Flash** — дешевле, но менее стабильный structured output и слабее в понимании кода
- **GPT-4.1-nano** — слишком низкое качество для генерации полноценных OpenAPI-спецификаций

#### Оценка стоимости

| Сценарий | Input токенов | Output токенов | Стоимость |
|---|---|---|---|
| Один файл (~200 строк) | ~3K | ~1.5K | ~$0.004 |
| Малый проект (5 файлов) | ~15K | ~3K | ~$0.01 |
| Средний проект (20 файлов) | ~60K | ~5K | ~$0.03 |
| 1000 запросов/день (средний) | ~60M | ~5M | ~$32/день |

### Можно ли решить без ML?

Частично — для конкретного фреймворка можно написать парсер (как swag для Go). Но универсальное решение для произвольного кода на разных языках без ML нереализуемо.

### Ограничения модели

- **Галлюцинации** — LLM может придумать несуществующие эндпоинты или параметры, особенно если код неоднозначен
- **Контекстное окно** — хотя 1M токенов это много, очень крупные монорепозитории могут не поместиться целиком; требуется предварительная фильтрация файлов
- **Детерминированность** — при одинаковых входных данных LLM может генерировать немного разные спецификации (митигируется через `temperature: 0` и structured output)
- **Зависимость от провайдера** — при недоступности OpenAI API сервис не сможет работать; митигируется абстракцией LLM-клиента для возможности переключения на другого провайдера
- **Качество на редких фреймворках** — модель лучше работает с популярными фреймворками (Gin, FastAPI, Express) и хуже с нишевыми или кастомными решениями

## Ограничения и нефункциональные требования

### Ограничения

- Максимальный размер запроса — 50 файлов или 500 KB суммарно
- Поддерживаемые языки в MVP: Go, Python, JavaScript/TypeScript
- Генерация только OpenAPI 3.0 (не Swagger 2.0, не AsyncAPI)
- Требуется доступ к OpenAI API (не работает офлайн)

### Нефункциональные требования

- **Производительность:** P95 ≤ 30 сек на запрос (включая время ответа LLM)
- **Доступность:** сервис отвечает на health check, graceful shutdown
- **Безопасность:** входные данные (исходный код) не сохраняются на диске и не логируются; API-ключ OpenAI хранится только в переменных окружения
- **Масштабируемость:** stateless-дизайн, горизонтальное масштабирование через добавление инстансов
- **Наблюдаемость:** структурированные логи (JSON), метрики latency и количества запросов

## Метрики качества

- **Валидность спецификации** — процент генераций, прошедших валидацию OpenAPI 3.0
- **Полнота** — recall: доля найденных эндпоинтов от реально существующих
- **Точность** — precision: доля корректных эндпоинтов среди найденных
- **Качество описаний** — ручная оценка по шкале 1–5
- **Latency** — P50, P95, P99 времени ответа
- **Стоимость** — средняя стоимость одного запроса в токенах и долларах

## MVP

### Что входит
- Анализ кода на Go, Python, JavaScript
- Генерация OpenAPI 3.0 (YAML/JSON)
- Один эндпоинт `/api/v1/analyze`
- Health check
- Docker-контейнеризация

- Предложения по улучшению API-дизайна
- Поддержка gRPC/GraphQL
- Web-интерфейс с визуализацией спецификации
- Кеширование результатов
- Аутентификация пользователей

## Масштабируемость

### Где узкое место?

Главное узкое место — **вызов OpenAI API**. Время ответа LLM составляет 5–25 секунд в зависимости от объёма кода, и это доминирует в общем latency запроса. Сам Go-сервис обрабатывает парсинг и сборку спеки за миллисекунды.

### Как масштабировать?

- **Горизонтально:** сервис stateless — достаточно запустить несколько инстансов за балансировщиком (nginx, k8s). Каждый инстанс независимо обращается к OpenAI API
- **Rate limiting на уровне LLM:** OpenAI имеет лимиты на RPM (requests per minute) и TPM (tokens per minute). При масштабировании нужно учитывать тарифный план и запрашивать повышение лимитов
- **Кеширование:** для повторных запросов с идентичным кодом можно кешировать результат (Redis), что снизит нагрузку на LLM и стоимость
- **Асинхронная обработка:** для крупных проектов — перевод на модель с очередью (запрос → job ID → poll результат)

### Какие риски при масштабировании?

- Рост стоимости пропорционален количеству запросов (нет economy of scale при использовании внешнего API)
- Rate limits OpenAI API могут стать блокером при высокой нагрузке
- При кешировании нужно учитывать инвалидацию (код меняется)

## Риски

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| LLM галлюцинирует несуществующие эндпоинты | Средняя | Высокое | Валидация спеки, сверка с AST где возможно |
| Большие проекты не влезают в контекст LLM | Средняя | Среднее | Разбивка на файлы, приоритизация роутов/хендлеров |
| Высокая стоимость при большом количестве запросов | Низкая | Среднее | Кеширование, выбор модели по сложности задачи |
| Зависимость от внешнего API | Средняя | Высокое | Абстракция LLM-клиента, возможность смены провайдера |
