# User Stories — AutoDoc API

## User Story 1 — Анализ исходного кода

**Как** backend-разработчик,
**я хочу** отправить исходный код моего сервиса и получить OpenAPI-спецификацию,
**чтобы** не писать документацию вручную.

### Критерии готовности
- `POST /api/v1/analyze` принимает JSON с массивом файлов (имя + содержимое)
- Сервис определяет язык и фреймворк
- Возвращается валидная OpenAPI 3.0 спецификация
- Возвращается краткое описание (summary) найденных эндпоинтов
- При невалидном запросе возвращается ошибка с описанием

---

## User Story 2 — Выбор формата спецификации

**Как** разработчик,
**я хочу** получить спецификацию в формате YAML или JSON на выбор,
**чтобы** использовать её в своём рабочем процессе без конвертации.

### Критерии готовности
- В запросе можно указать `"format": "yaml"` или `"format": "json"`
- По умолчанию возвращается YAML
- Спецификация корректна в обоих форматах

---

## User Story 3 — Анализ кода на разных языках

**Как** тимлид с проектами на разных языках,
**я хочу** анализировать код на Go, Python и JavaScript,
**чтобы** получить единообразную документацию для всех сервисов.

### Критерии готовности
- Поддерживаются Go (net/http, Gin), Python (FastAPI, Flask), JavaScript (Express)
- Язык определяется автоматически или указывается в запросе
- Качество спецификации сопоставимо для всех поддерживаемых языков

---

## User Story 4 — Проверка здоровья сервиса

**Как** DevOps-инженер,
**я хочу** проверять доступность сервиса через health check,
**чтобы** мониторить его состояние.

### Критерии готовности
- `GET /api/v1/health` возвращает `200 OK` с информацией о статусе
- Ответ включает версию сервиса и статус подключения к LLM API

---

## User Story 5 — Обработка ошибок

**Как** пользователь API,
**я хочу** получать понятные сообщения об ошибках,
**чтобы** понимать, что пошло не так.

### Критерии готовности
- Пустой запрос → 400 с описанием
- Слишком большой код (> лимита контекста) → 413 с рекомендацией
- Ошибка LLM API → 502 с описанием
- Невалидный формат → 422 с описанием
