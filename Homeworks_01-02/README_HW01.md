# MealMate — трекер питания

Пункт 1. Концепция продукта

### 1) Коротко об идее

MealMate помогает пользователю быстро заносить съеденные продукты и мгновенно видеть дневную сумму калорий и БЖУ. Основа — справочник продуктов с питательностью «на 100 г» или «на порцию», логирование приёмов пищи и агрегирование по локальному дню пользователя.

### 2) Жизненный цикл продукта (от идеи до поддержки)

#### Этап 0 — Discovery (1–2 недели)
- Цель: подтвердить потребность: «быстрый лог + понятная сводка по дню».
- Активности: интервью (5–10 пользователей), скетчи UX, приоритизация требований.
- Артефакты: Problem Statement, список user stories, черновой бэклог.
- Критерий выхода: сформулирован MVP и метрики успеха.

#### Этап 1 — MVP (2–3 спринта по 1 неделе)
- Объём: базовый справочник, лог приёмов пищи, сводка за день, минимальные тесты.
- Техчасть: монолит FastAPI, SQLite, контейнеризация.
- Метрики: DAU, среднее число логов в день, время создания записи (< 2 сек).
- Критерий выхода: ≥70% пользователей создают ≥2 записи в первый день.

#### Этап 2 — Beta (2–4 недели)
- Объём: рецепты (композиция продуктов), экспорт/импорт данных, упрощённая авторизация.
- Техчасть: миграции БД, базовые метрики и логирование.
- Метрики: удержание D7, средние отклонения подсчётов (валидаторы БЖУ).
- Критерий выхода: стабильные метрики производительности и удовлетворённости.

#### Этап 3 — GA / Growth
- Объём: теги, быстрые пресеты, простая аналитика за неделю/месяц.
- Техчасть: вынос справочника в отдельный модуль/контекст, кеширование.
- Метрики: MAU, конверсия в «дневную цель закрыта», NPS.

#### Этап 4 — Поддержка и эволюция
- Активности: устранение дефектов, оптимизация, расширение API.
- Эксплуатация: регулярные минорные релизы, мониторинг SLO (доступность, латентность).
- Консервация/депрекация: план вывода фич/версий (sunset policy).

### 3) Ключевые требования к системе

#### Функциональные (MVP)
##### 1. Справочник продуктов
- Создание продукта с питательностью: на 100 г или на порцию (с serving_grams).
- Просмотр/поиск по названию.
##### 2. Лог приёма пищи
- Добавить запись: продукт + либо grams, либо servings (ровно одно).
- Указание consumed_at (по умолчанию — сейчас).
##### 3. Сводка за день
- Суммы kcal, protein, fat, carb за локальный день пользователя.
- Список записей дня.
##### 4. Валидации
- Положительные массы/порции; нутриенты неотрицательны.
- Предупреждение, если kcal сильно расходится с формулой 4/9 (Б/У×4 + Ж×9).
##### 5. Наблюдаемость (observability)
- Логи создания записей; счётчик meals_logged; простая метрика латентности.

#### Нефункциональные
- Производительность: P95 на POST/GET ≤ 300 мс при 50 rps (MVP-порог).
- Надёжность: сохранение записей без потерь; откат транзакций при ошибке.
- Безопасность: базовая авторизация (токен/JWT) к своему ресурсу (в MVP — опционально).
- Скалируемость: моно-репо/монолит, готовность к горизонтальному масштабированию API.
- Сопровождаемость: слои (API → Service → Domain → Repo), типизация, тесты.
- Портируемость: контейнеризация (Docker), dev/QA/prod профили.

#### Ограничения и допущения
- На MVP — одна временная зона на пользователя (хранится у пользователя).
- Источник справочника — вручную/импорт CSV; интеграции с внешними базами позже.
- Нет сложной аналитики/рекомендаций в MVP.

### 4) DDD: контексты, модель и связи
#### 4.1 Bounded Contexts
##### 1. User (Пользователь)
- Ответственность: профиль, локальный TZ, владение данными.
- Основные сущности: User.
- Интерфейсы: REST (/users/*), выдаёт user_id, timezone.
- Зависимости: не зависит от домена питания.
##### 2. Nutrition (Питание / Справочник)
- Ответственность: продукты и (в будущем) рецепты, их питательность.
- Сущности/агрегаты:
  - Food (агрегат): id, name, mode (per_100g/per_serving), нутриенты, serving_grams?.
  - Recipe (позже): композиция RecipeItem(food_id, grams), расчёт сводной питательности.
- Инварианты:
  - Нутриенты ≥ 0; для per_serving указан serving_grams.
- Интерфейсы: /foods/* (CRUD/поиск).
- Особенности: может со временем стать отдельным микросервисом.
##### 3. Logging (Журналирование приёмов пищи)
- Ответственность: создание и хранение логов приёмов пищи для пользователя.
- Сущности/агрегаты:
  - MealLog (агрегат): id, user_id, food_id, (grams XOR servings), consumed_at.
- Инварианты:
  - Ровно один из grams или servings > 0.
  - food_id валиден в Nutrition.
- Доменная логика: расчёт фактических БЖУ/ккал для записи на лету (по «правилу источника истины» — из Nutrition).
- Интерфейсы: /meals/*.
##### 	4.	Summary (Сводки/Агрегация)
- Ответственность: агрегирование по локальному дню пользователя.
- Сущности (вычислимые модели): DailySummary (не хранится, формируется запросом).
- Интерфейсы: /summary?date=YYYY-MM-DD.
- Зависимости: читает MealLog (Logging) и Food (Nutrition); учитывает TZ из User.

### 5) Выбор архитектуры и обоснование
#### Выбор: модульный монолит (FastAPI + одна БД).
Для MVP это быстрее и надёжнее: одна кодовая база, единые миграции и транзакции, простая отладка и деплой. Лишних накладных расходов на сетевые вызовы и согласование данных нет, проще держать инварианты домена.

Минусы: хуже независимое масштабирование и релиз отдельных частей; если нагрузка вырастет, выделим контексты Nutrition и Summary в отдельные сервисы (естественный путь к микросервисам).


### User stories (ориентир — MVP MealMate)
#### User Story 1 — добавление продукта
Как пользователь, я хочу добавить продукт с указанием калорийности (на 100 г или на порцию), чтобы потом использовать его при фиксации приёмов пищи.
##### Критерии готовности:
- POST /foods принимает FoodCreate, валидирует поля.
- В базе создаётся запись; на ответе возвращается id, name, mode.
- Ошибка при дубликате имени.

#### User Story 2 — добавление приёма пищи
Как пользователь, я хочу записывать, что и сколько я съел, чтобы MealMate мог подсчитать мой дневной рацион.
##### Критерии готовности:
- POST /meals принимает MealCreate.
- Валидаторы проверяют: одно из grams / servings, значения > 0.
- В базе создаётся MealLog.
- Возвращается MealOut с расчётом калорий.

#### User Story 3 — просмотр дневной сводки
Как пользователь, я хочу видеть суммарное количество калорий за день, чтобы контролировать питание.
##### Критерии готовности:
- GET /summary?day=YYYY-MM-DD
- Возвращает total kcal и список приёмов за день.
- Работает корректно при разных часовых поясах (UTC).
